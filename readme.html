https://jimmymega.shinyapps.io/app_vademecum/ # Veterinary Knowledge Suite *A metadata-driven engine for building veterinary knowledge applications.* ------------------------------------------------------------------------ # Overview Veterinary Knowledge Suite is a generic knowledge engine designed to organize, navigate and present structured veterinary information. Unlike traditional applications, the engine is completely independent of the knowledge domain. Pharmacology, Clinical Medicine, Surgery, Microbiology, Zoology or any other discipline are defined exclusively by the external data source. The application interprets metadata rather than hard-coded knowledge, allowing new disciplines to be incorporated without modifying the source code. ------------------------------------------------------------------------ # Project philosophy The engine has been designed following a small set of architectural principles intended to maximize scalability, maintainability and long-term usability. ## Principle I — Data governs behavior The structure of the application is entirely defined by the external data source. The code never imposes a hierarchy; instead, it interprets the metadata contained in the workbook. ------------------------------------------------------------------------ ## Principle II — Domain independence The application contains no veterinary knowledge. All discipline-specific information resides exclusively in the external data source. The same engine can therefore support pharmacology, surgery, microbiology, parasitology, zoology or any other structured knowledge base. ------------------------------------------------------------------------ ## Principle III — Metadata-driven architecture Each column receives its role through metadata. Adding or removing content requires only modifications to the workbook. The application code remains unchanged. ------------------------------------------------------------------------ ## Principle IV — Single responsibility Every helper function performs one clearly defined task. Functions remain independent and understandable without requiring knowledge of the entire project. ------------------------------------------------------------------------ ## Principle V — Server as orchestrator The server contains only the reactive workflow. Business logic, navigation and presentation are delegated to helper functions. ------------------------------------------------------------------------ ## Principle VI — Configuration over programming Adding new hierarchy levels, panels or resources requires editing the workbook rather than modifying the application code. ------------------------------------------------------------------------ ## Principle VII — Complementary navigation strategies The engine simultaneously supports two navigation paradigms. ### Hierarchical exploration Designed for learning. Users progressively navigate from general concepts toward specific records. Example: System ↓ Subsystem ↓ Drug class ↓ Drug subgroup ↓ Active ingredient ### Incremental search Designed for rapid consultation. Users search directly for an active ingredient, indication or other indexed term. Both navigation strategies converge to a single active record. ------------------------------------------------------------------------ ## Principle VIII — Readability over abstraction Explicit, self-contained code is preferred over aggressive factorization. Small, understandable functions simplify maintenance over the lifetime of the project. ------------------------------------------------------------------------ ## Principle IX — Scalable architecture Future migration from Excel to Google Sheets, Supabase, PostgreSQL or SQL Server should require modifications only to the data access layer. The navigation and presentation engines remain unchanged. ------------------------------------------------------------------------ # System architecture ``` External data source │ ▼ Metadata │ ▼ Navigation engine │ ▼ Active record │ ▼ Presentation engine │ ▼ User interface ``` ------------------------------------------------------------------------ # Repository structure ``` VeterinaryKnowledgeSuite/ │ ├── app.R ├── config.R ├── ui.R ├── server.R │ ├── block_1.R ├── block_2.R ├── block_3.R ├── block_4.R ├── block_5.R ├── block_6.R │ ├── R/ │ actualizar_picker.R │ buscar_registros.R │ resolver_registro.R │ construir_accordion.R │ construir_encabezado.R │ construir_recursos.R │ contenido_panel.R │ columnas_estructura.R │ columnas_paneles.R │ componentes.R │ obtener_registro.R │ opciones_busqueda.R │ primer_texto.R │ texto_lista.R │ titulo.R │ valores_disponibles.R │ └── data/ info.xlsx ``` ------------------------------------------------------------------------ # Knowledge source The engine is entirely controlled by an external workbook. Currently two sheets are used. | Sheet | Purpose | |---------|----------------------------------------------| | General | Main knowledge base | | Dosis | Drug dosage database (future implementation) | ------------------------------------------------------------------------ # Workbook structure The workbook contains two metadata rows followed by the records. | Row | Purpose | |-------|----------------------| | 1 | Column type | | 2 | Internal column name | | 3...n | Records | Example | structure | structure | structure | panel | panel | resource | |-----------|------------|------------------|-------|-----------|----------| | sistema | subsistema | principio_activo | usos | toxicidad | video | | Nervioso | Autónomo | Betanecol | ... | ... | ... | ------------------------------------------------------------------------ # Metadata The first row defines how each column behaves. Typical values include - structure - panel - resource - search - index The engine interprets these values dynamically. ------------------------------------------------------------------------ # Structural columns Structural columns define the conceptual hierarchy. Their order determines the navigation sequence. For example ``` Sistema ↓ Subsistema ↓ Grupo farmacológico ↓ Subgrupo ↓ Principio activo ``` No hierarchy is hard-coded. Changing the order of the columns changes the hierarchy. ------------------------------------------------------------------------ # Panel columns Panel columns contain the knowledge associated with each record. Examples - General description - Clinical use - Contraindications - Pharmacokinetics - Adverse effects Only non-empty panels are displayed. ------------------------------------------------------------------------ # Resource columns Resource columns contain complementary information. Examples - Images - Videos - External links - References Resources appear only when available. ------------------------------------------------------------------------ # Navigation workflow Two independent navigation strategies converge to the same active record. ``` Hierarchical navigation ───────────────┐ │ ▼ Active record ▲ ───────────────┘ Incremental search ``` ------------------------------------------------------------------------ # Reactive workflow The server is organized into six independent blocks. ``` Block I Reactive state ↓ Block II Initialization ↓ Block III Navigation ↓ Block IV Active record ↓ Block V Presentation ↓ Block VI Interface synchronization ``` ------------------------------------------------------------------------ # Extending the engine Supporting a new discipline should require no changes to the application. Only the workbook is modified. Examples - Veterinary Pharmacology - Clinical Medicine - Surgery - Microbiology - Virology - Parasitology - Zoology - Toxicology - Animal Nutrition ------------------------------------------------------------------------ # Long-term vision Veterinary Knowledge Suite is not intended merely as a pharmacology application. Its long-term objective is to provide a unified platform capable of supporting multiple veterinary disciplines through a common metadata-driven engine. Students use the platform to explore concepts. Professionals use the same platform for rapid consultation. The knowledge evolves independently from the software. ------------------------------------------------------------------------ # Authors SJWatson Developed with assistance from OpenAI ChatGPT (GPT-5.5). ------------------------------------------------------------------------ # License To be defined. ------------------------------------------------------------------------ # Citation To be incorporated in future releases.
